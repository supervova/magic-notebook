# Сервис проверки практической стилистики — Спецификация

Статус: Draft / Архитектура утверждена

## Цель

Создать бесплатный браузерный сервис проверки текста, который:

- привлекает органический трафик на сайт Magic Notebook,
- демонстрирует экспертность продукта,
- в дальнейшем становится частью редактора Magic Notebook.

Сервис анализирует текст пользователя и подсвечивает проблемы практической стилистики.

Поддерживаемые языки в MVP:

- русский,
- английский.

Система должна работать без платных LLM API и без постоянных расходов на backend.

## Позиционирование продукта

Сервис — не AI-писатель. Сервис должен быть:

- быстрым,
- приватным,
- детерминированным,
- объяснимым,
- пригодным для офлайна,
- дешёвым в поддержке.

Основное позиционирование:

> Спокойные инструменты для письма. Без промптов. Без подписок. Без слежки.

## Архитектурное решение

### Главный принцип

Вся логика анализа текста должна жить в отдельном переиспользуемом пакете.

Веб-интерфейс и интеграция в Magic Notebook должны быть тонкими оболочками вокруг одного и того же движка.

## Архитектура верхнего уровня

### Текущая структура внутри Astro-сайта

На старте сервис разрабатывается внутри репозитория сайта Magic Notebook.

Astro использует file-based routing, поэтому файлы маршрутов остаются в `src/pages`.

Вся бизнес-логика сервиса должна жить отдельно от маршрутов:

```txt
src/
  pages/
    ru/
      text-checker.astro
    text-checker.astro

  templates/
    text-checker.astro

  features/
    text-checker/
      checks/
      components/
      core/
      dictionaries/
        en/
        ru/
      styles/
      types/
      utils/
      workers/
```

Разделение ответственности:

| Директория                  | Ответственность                          |
| --------------------------- | ---------------------------------------- |
| `src/pages`                 | Маршруты, SEO, meta, локализованные URL  |
| `src/templates`             | Композиция страницы и подключение layout |
| `src/features/text-checker` | Вся логика проверки текста и UI-фичи     |

Запрещено размещать tokenizer, checks, dictionaries и scoring прямо в `pages` или `templates`.

## Почему не LLM

LLM-подход отклонён для MVP, потому что:

- нет бюджета на API,
- неограниченные расходы при росте,
- высокая задержка,
- проблемы приватности,
- слабая детерминированность,
- сложное тестирование,
- сложная офлайн-интеграция в desktop-приложение.

Предпочтителен rule-based подход.

## Функциональность

### Phase 1 — Стоп-слова и канцелярит

Вдохновлено: [Главред](https://glvrd.ru/)

Функции:

- поиск стоп-слов: канцеляриты, штампы, лексические ошибки,
- поиск слабых конструкций,
- подсветка совпадений,
- объяснения,
- расчёт оценки текста.

Автоматический rewrite в MVP не входит.

### Phase 2 — Тавтология и плеоназмы

Вдохновлено: [Свежий взгляд](https://gitlab.com/opennota/fresheye)

Функции:

- повтор близко расположенных слов,
- повтор однокоренных слов,
- лексические дубли,
- словарь плеоназмов,
- повтор фраз.

### Phase 3 — Удобочитаемость

Вдохновлено: [Hemingway](https://hemingwayapp.com/readability-checker)

Функции:

- сложность предложений,
- поиск длинных предложений,
- эвристика passive voice,
- readability score,
- классификация сложности чтения.

Для английского:

- Flesch Reading Ease,
- Flesch-Kincaid.

Для русского:

- адаптированная readability-эвристика,
- эвристика длины предложений,
- эвристика слоговой сложности.

## Что НЕ входит в MVP

MVP не включает:

- генеративный rewrite,
- cloud sync,
- аккаунты,
- платные подписки,
- совместное редактирование,
- AI-чат,
- embeddings,
- vector databases,
- grammar correction,
- plagiarism detection.

## Основные принципы

### Детерминированность

Одинаковый ввод всегда даёт одинаковый результат.

### Объяснимость

Каждое предупреждение должно содержать:

- тип проблемы,
- объяснение,
- при необходимости — рекомендацию.

### Скорость

Цели:

| Метрика             | Цель                       |
| ------------------- | -------------------------- |
| Первый анализ       | < 100ms для средней статьи |
| Размер бандла       | < 500KB gzip на старте     |
| Зависимость от сети | Нет                        |

### Privacy-first

- Вся обработка выполняется в браузере.
- Текст не хранится.
- Удалённый анализ отсутствует.

## API общего ядра

### Входные данные

```ts
interface AnalyzeTextInput {
  text: string;
  language?: 'ru' | 'en';
  checks?: CheckId[];
}
```

### Результат

```ts
interface AnalyzeTextResult {
  score: number;
  issues: TextIssue[];
  statistics: TextStatistics;
}
```

### Модель проблемы

```ts
interface TextIssue {
  id: string;
  checkId: CheckId;
  severity: 'info' | 'warning' | 'danger';
  title: string;
  description: string;
  suggestion?: string;
  startIndex: number;
  endIndex: number;
  replacements?: string[];
}
```

### Модель статистики

```ts
interface TextStatistics {
  characters: number;
  words: number;
  sentences: number;
  paragraphs: number;
  readabilityScore?: number;
}
```

### Идентификаторы проверок

```ts
export type CheckId =
  | 'stop-words'
  | 'bureaucracy'
  | 'cliches'
  | 'tautology'
  | 'pleonasm'
  | 'readability';
```

## Стратегия токенизации

Нужен лёгкий tokenizer.

Требования:

- поддержка Unicode,
- поддержка кириллицы,
- разделение пунктуации,
- сегментация предложений,
- сегментация абзацев.

Тяжёлые NLP-frameworks в MVP избегаются.

## Система оценки

Начальная модель:

```txt
100
- штрафы за стоп-слова
- штрафы за канцелярит
- штрафы за readability
```

Ограничение:

```txt
0..100
```

Интерпретация:

| Оценка | Значение            |
| ------ | ------------------- |
| 90-100 | Отлично             |
| 70-89  | Хорошо              |
| 50-69  | Нужна редактура     |
| <50    | Текст тяжело читать |

## Стратегия развития

### Этап 1 — MVP в Astro-сайте

Сервис реализуется внутри репозитория сайта. Причины:

- это публичная SEO-страница,
- быстрее проверить спрос,
- проще подключить к текущей локализации сайта,
- не нужно заводить третий репозиторий.

### Этап 2 — Стабилизация ядра

После появления рабочих проверок нужно стабилизировать:

- публичный API анализа,
- формат `TextIssue`,
- формат словарей,
- scoring,
- тестовые fixtures.

### Этап 3 — Выделение shared package

Когда API перестанет часто меняться, логика переносится из репозитория сайта в репозиторий приложения:

```txt
site/src/features/text-checker/core
site/src/features/text-checker/checks
site/src/features/text-checker/dictionaries
↓
app/packages/text-checker-core
app/packages/text-checker-dictionaries
```

Сайт после этого использует собранный пакет или синхронизированную сборку.

### Этап 4 — Интеграция в Magic Notebook

Тот же движок подключается к редактору Magic Notebook. Интеграция не должна дублировать правила, словари и scoring.

## Будущая интеграция в Magic Notebook

Движок должен позже поддерживать:

- live-анализ текста,
- inline-подсветку,
- панель рекомендаций,
- writing assistant mode,
- офлайн-работу внутри desktop-приложения.

Точка интеграции:

```txt
Magic Notebook editor
  → text-checker-core
```

Без дублирования логики.

## Риски

### False Positives

Rule-based системы могут слишком агрессивно подсвечивать текст. Смягчение:

- уровни severity,
- отключаемые проверки,
- консервативные значения по умолчанию.

### Сложность русской морфологии

Русские склонения и формы слов сложны. Смягчение:

- сначала простые эвристики,
- dictionary-first подход,
- optional stemming позже.

### Рост размера бандла

Большие словари могут раздуть bundle. Смягчение:

- lazy-loading языковых пакетов,
- сжатие словарей,
- split по функциональности.
